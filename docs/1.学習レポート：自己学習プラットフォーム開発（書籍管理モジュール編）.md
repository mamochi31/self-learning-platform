# 学習レポート：自己学習プラットフォーム開発（書籍管理モジュール編）

本レポートは、Next.js + NestJS + Prisma 7 を用いたモノリポ環境での「書籍管理機能」のバックエンド実装を通じて習得した技術要素と設計思想をまとめたものです。

---

## 1. システム構成と設計思想

### 1.1 アーキテクチャ概略
* **モノリポ構成 (Turborepo)**: フロントエンド、バックエンド、共有パッケージを一元管理し、開発効率を最大化。
* **ドメイン駆動設計 (DDD)**: ビジネスルールを技術詳細から分離し、長期的なメンテナンス性を確保。
* **最新環境の採用**: Node.js v22 および TypeScript 7.0 (移行準備) 基準の設定を導入。

### 1.2 DDD 4層レイヤーの役割
1.  **Domain (ドメイン層)**: `Book` エンティティ（ビジネスルール）と `IBookRepository`（インターフェース）を定義。技術への依存を一切排除。
2.  **Application (アプリケーション層)**: `CreateBookUseCase` が業務フローを指揮。ドメインオブジェクトを操作する「指揮者」。
3.  **Infrastructure (インフラ層)**: `PrismaBookRepository` が DB 操作を担当。Prisma 7 のアダプター方式を実装。
4.  **Presentation (プレゼンテーション層)**: `BookController` が HTTP リクエストを受け付け。

## 2. NestJS の高度な技術要素

### 2.1 依存性の注入 (DI) と Symbol
* **インターフェースの型消失**: TypeScript のインターフェースは実行時に消滅するため、DI トークンとして `Symbol` を利用。
* **依存性の逆転 (DIP)**: 具体的な実装（Prisma）ではなく抽象（インターフェース）に依存させることで、疎結合な設計を実現。

### 2.2 デコレータの役割
* **@Injectable()**: NestJS の DI コンテナにクラスを「部品」として登録。
* **@Controller()**: ルーティング機能と DI コンテナへの登録を兼務。
* **@Inject()**: Symbol トークンを指定して、インターフェースに基づいた部品を注入。

## 3. Prisma 7 と Node.js v22 への適応

### 3.1 接続方式の刷新
* **datasourceUrl の廃止**: Prisma 7.0.0 以降の仕様変更に合わせ、`pg` アダプターを用いたドライバーアダプター方式へ移行。
* **カプセル化**: モノリポのホーイスティング問題を避けるため、Prisma Client をパッケージ内の `generated/client` に強制出力。

### 3.2 ESM 解決ルールの遵守
* **拡張子問題**: Node.js v22 の厳格なルールに基づき、インポートパスへの `.js` 追加や、ディレクトリインポートの回避を実施。

## 4. トラブルシューティングの実績

| 課題 | 主な原因 | 解決策 |
| :--- | :--- | :--- |
| **404 Not Found** | Turborepo 内のポート競合 | 競合アプリ（docs）の停止と api ポートの固定 |
| **MODULE_NOT_FOUND** | ビルド結果（dist）の階層構造不一致 | `rootDir` の設定と `nest-cli.json` の調整 |
| **TS5095 (TS 設定競合)** | module と moduleResolution の不整合 | Node16 設定への統一と CommonJS モードの採用 |
| **DB Access Denied** | 環境変数の未ロード・ユーザー名不足 | `pg.Pool` での `dotenv/config` 明示と `.env` 配置 |
