# NestJS 実践学習レポート：DDD 設計と最新技術への適応

本レポートは、自己学習プラットフォーム開発における NestJS バックエンド実装を通じて習得した設計思想と技術的解決策をまとめたものです。

---

## 1. DDD（ドメイン駆動設計）に基づく 4 層レイヤーの実装

ビジネスルールを技術詳細から保護するため、NestJS のモジュールシステムを利用して以下の 4 レイヤーを構築しました。

### 1.1 各レイヤーの責務

**Domain (ドメイン層):**
*   `Book` エンティティを定義。タイトルの必須チェック等のビジネスルールをクラス内にカプセル化。
*   `IBookRepository` インターフェースを定義し、永続化の方法を抽象化。

**Application (アプリケーション層):**
*   `CreateBookUseCase` を実装。ドメインモデルの生成とリポジトリへの保存という業務フローをオーケストレート（指揮）する。

**Infrastructure (インフラ層):**
*   `PrismaBookRepository` を実装。Prisma 7 の「ドライバーアダプター」を介して PostgreSQL と通信。
*   `PrismaService` で DB 接続のライフサイクルを管理。

**Presentation (プレゼンテーション層):**
*   `BookController` を実装。HTTP リクエストを受け取り、UseCase へ処理を委譲。

## 2. NestJS における依存性注入 (DI) の設計

### 2.1 Symbol によるインターフェース注入

TypeScript のインターフェースは実行時に消失するため、`Symbol` を DI トークンとして活用しました。これにより、ドメイン層が具体的な実装に依存しない **依存性の逆転 (DIP)** を実現しました。

### 2.2 デコレータの使い分け

*   `@Injectable()`: クラスを NestJS の部品（Provider）として登録。
*   `@Inject()`: インターフェースの代わりに `Symbol` を指定して部品を注入。
*   `@Controller()`: ルーティングと DI 管理の両方の役割を付与。

## 3. Prisma 7 と Node.js v22 への適応

### 3.1 接続方式の刷新

Prisma 7 で廃止された `datasourceUrl` の代わりに、`@prisma/adapter-pg` と `pg.Pool` を用いたドライバーアダプター方式を導入しました。これにより、Node.js ランタイム上で直接接続を管理するモダンな構成となりました。

### 3.2 モジュール解決の最適化

Node.js v22 の ESM 解決ルールに対応するため、`Node16` ベースのモジュール解決を採用し、パスのエイリアス（`@/`）や共有パッケージ（`@repo/database`）の型解決を正常化しました。

## 4. トラブルシューティングの記録

| 課題 | 原因 | 解決策 |
| :--- | :--- | :--- |
| API への 404 エラー | Turborepo 内のポート重複 (docs) | 競合プロセスの停止とポートの固定 |
| `MODULE_NOT_FOUND` | ビルド出力階層 (`dist`) の不一致 | `rootDir: "."` の設定による階層固定 |
| `P1012` (DB 接続エラー) | 環境変数読み込みの順序問題 | `dotenv/config` の明示的読み込みと Pool への注入 |

## 5. 結論

今回の実装により、NestJS を単なるフレームワークとしてではなく、ドメインの整合性を守るための「堅牢な設計ツール」として活用する手法を習得しました。これが今後の機能拡張における強力な土台となります。
